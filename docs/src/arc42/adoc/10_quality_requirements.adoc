== Quality Requirements

=== Quality Requirements Overview

The quality goals (Chapter 1) derive from the laws (Chapter 2):

[options="header"]
|===
| Attribute | Requirement
| *Integrity* | Every accepted byte is re-derived against its κ (Law L5).
| *Reproducibility* | A holospace's identity is the κ of its definition; identical inputs yield identical holospaces.
| *Portability* | The same holospace κ boots on any peer (browser / native / bare-metal).
| *Efficiency (memory)* | The store is the address space; content dedupes; RAM is a bounded cache (Law L3).
| *Autonomy (no server)* | No host, no account, no control plane; peers are content-addressed (Law L1).
| *Authority* | The documentation is the single authoritative source; features trace to it.
|===

=== Quality Scenarios

[options="header"]
|===
| # | Scenario | Expected response
| QS1 | The same git repo + `devcontainer.json` is provisioned twice, on different peers. | Both yield the same holospace κ.
| QS2 | A holospace is suspended on one signed-in instance and resumed on another. | It resumes from the κ snapshot; unchanged state dedupes.
| QS3 | A gateway (e.g. GitHub Pages, or a peer) returns bytes that do not match the requested κ. | The bytes are rejected on re-derivation; the fetch fails over to another source.
| QS4 | A holospace's total state exceeds available RAM. | It still boots; content is demand-paged by κ-resolve, evicted by garbage collection.
| QS5 | An operator signs in on a new instance. | Their holospaces and state are discoverable and synchronise over the substrate.
| QS6 | A contributor implements a feature. | The feature links to the documentation section it realizes; the V1–V8 pipeline passes.
| QS7 | An unmodified Linux devcontainer is provisioned and booted. | Its real operating system boots to userspace and runs its real binaries (a shell, `apt`); behaviour matches the Linux specification.
| QS8 | An operator edits a file and runs a terminal command in a holospace's workspace projection. | The edit and command publish as canonical events on the holospace's channels; the κ snapshot advances; the editor and file tree reflect the new content (read by κ).
| QS9 | An operator signs in to the Platform Manager and organizes their holospaces, performing lifecycle operations (provision · enter · suspend · resume · terminate). | The console authenticates a self-sovereign identity, lists and organizes the holospaces with status, and each operation runs through the real browser peer; the console holds no state of its own (the holospaces and roster are κ in the store).
| QS10 | An operator *enters* a holospace from the Platform Manager. | A new browser tab launches the VS Code experience — the real Monaco editor and xterm.js terminal, integrity-verified — over the running devcontainer: the terminal drives the booted OS, the editor and file tree read/edit content by κ.
| QS11 | The devcontainer boots its operating system. | A real distribution root filesystem mounts over a VirtIO block device from the κ-disk; the terminal runs the real binaries (a shell, the coreutils) — behaviour matching the Linux specification and QEMU.
| QS12 | The operator edits a file in the editor and reads it in the terminal. | The editor and the OS share the workspace filesystem over VirtIO-9p (9P2000.L); the edit (content by κ) is the same file the terminal reads — one content, no copy.
| QS13 | The devcontainer runs its `devcontainer.json` `features` / `postCreateCommand`. | The OS reaches the network over VirtIO-net and the package manager installs the real packages; the setup commands run, matching the Dev Container specification.
| QS14 | The operator uses the workspace's language intelligence and extensions. | The real VS Code workbench (κ-verified) speaks the Language Server and Debug Adapter protocols to tools running in the devcontainer OS, and runs extensions the `devcontainer.json` selects — the authentic Codespaces/Gitpod experience.
|===

=== Verification and Validation

holospaces is evaluated against *external authoritative specifications and standards* — never against itself (no self-reference). The V&V is *defined by this documentation* and is authoritative: each invariant below names the external authority it is validated against, the mechanism that enforces it, and the witness. Real validation artifacts are imported and pinned, with provenance recorded in `vv/PROVENANCE.md`; imported artifacts are themselves content-addressed and verified on import. The executable framework lives in `vv/` and is the single V&V entry point (`vv/run.sh`).

The framework has two tiers:

* *Specification conformance* (`CS`) — the holospaces specification (this documentation) validated against the standards it is authored under. Live on every build.
* *Component conformance* (`CC`) — each component validated against the external authority for its behavior. Each is specified here and witnessed by its suite against that authority, never by self-reference.

| CC-25 | The devcontainer's Dev Container Features install on create | the https://containers.dev/implementors/features[Dev Container Features] specification (the feature artifact format — an OCI artifact carrying `devcontainer-feature.json` + `install.sh` — and the install contract: `install.sh` runs in the container with the declared options as environment, before the lifecycle commands) and the `ext4` on-disk format (e2fsprogs as the oracle), with `qemu-system-riscv64` as the differential runtime oracle | the Boot Orchestrator parses the `features` a `devcontainer.json` declares (`CC-4`), **imports each feature artifact by κ** (verify-by-re-derivation — the `CC-20` machinery, *no notional functionality*) and **places it into the rootfs**, and the generated `/init` runs each feature's `install.sh` *before* the lifecycle commands (`CC-22`), with the declared options passed as *uppercased environment* — so feature-provided tools are present on entry, exactly as a Codespace installs features (ADR-016). Not parse-and-drop: the declared feature is *applied* | live (`vv/suites/cc25-features`)
| CC-26 | A devcontainer declared with a Dockerfile build is built on the substrate | the https://docs.docker.com/reference/dockerfile/[Dockerfile reference] (the instruction semantics — `FROM`/`ARG`/`ENV`/`WORKDIR`/`COPY`/`RUN`; line continuations; variable substitution) and the Dev Container spec's `build`, with the `ext4` format (e2fsprogs) and `qemu-system-riscv64` as the differential oracle | a `devcontainer.json` may declare its container as a *Dockerfile build* (`"build": { "dockerfile": … }`) instead of a prebuilt `image`; holospaces parses it (`CC-4`, retained as `ImageSource::Build`, *not dropped*) and honours it the substrate-native way — *no Docker daemon*: the Dockerfile's `FROM` image is pulled + assembled as the base rootfs (`CC-20`/`CC-10`), the `COPY` sources from the build context are injected into the rootfs, and the `RUN` instructions run **in the devcontainer OS** during the build phase with the build `ARG`s/`ENV`s in scope (`CC-22`/`CC-25` machinery) — the built rootfs is the result. An unknown Dockerfile instruction is an explicit error, never a silent drop | live (`vv/suites/cc26-build`)
| CC-27 | A devcontainer declared with a Docker Compose file resolves its service | the https://compose-spec.io[Compose specification] (the `services.<name>.{image\|build}` model) and the Dev Container spec's `dockerComposeFile` + `service` | a `devcontainer.json` may declare its container with a `dockerComposeFile` and a `service` (the service that *is* the devcontainer); holospaces parses it (`CC-4`, retained as `ImageSource::Compose`, *not dropped*), reads the compose file from the repository, and resolves the named service's image source — its `image` (pulled, `CC-20`) or its `build` (a Dockerfile build, `CC-26`) — and provisions from it. A missing or ambiguous service is an explicit error, never a silent default; multi-service orchestration is out of scope for a single devcontainer | live (`vv/suites/cc27-compose`)
| CC-28 | The control plane reconfigures a running instance over the substrate | the substrate's content-addressing + sync contract (a real loopback HTTP content-addressed gateway, *verify-on-receipt* — Law L5, the model the `e2e` roster-sync witness uses) and, for the live network directive, a real host `TcpListener` on the running machine (the `CC-21` ingress dual); the Codespaces/Gitpod (Ona) control-panel *reconfigure* UX | the control panel reconfigures a *running* instance across the four operation classes — *lifecycle / storage / network / account-user* — UOR-native (ADR-018): the panel builds a κ-addressed `Configuration` (a hologram `Realization` embedding the issuing operator identity and the target instance κ, plus ordered directives), `Manager::configure` stores it (Law L2) and records it as the instance's current configuration, and the running instance *resolves* it over the substrate (verify-by-re-derivation, Law L5), checks it targets *this* instance and the operator is authorized, and *applies* it. The control plane **actually manages** the instance: `Manager::reconfigure` publishes the configuration *and drives the running session with it* — a lifecycle directive really transitions it (suspend → κ snapshot, resume, terminate), and a storage/network directive replaces the effective capability set (a new κ, persisted). A `forwardPort` directive forwards a port on the *running* machine (`Machine::forward_port`, a real reachable listener bound live — no reboot). In the browser, the panel carries the configuration to the running instance tab over a `BroadcastChannel` control channel, which verifies it (L5) and applies it (`Workspace::reconfigure`). Reproducible (same configuration ⇒ same κ, Law L1) and authority-scoped (a configuration for a *different* instance, or from an *unauthorized* operator, is refused — Law L3). No server, no control-plane→instance RPC; the configuration is content, the substrate carries it | live (`vv/suites/cc28-reconfigure`)
| CC-29 | L5 verification is placed at the trust boundary, not charged on every local read | the substrate's own `verify_kappa` re-derivation contract — the exact primitive `get_with_fetch` (the loopback HTTP-CAS gateway, *verify-on-receipt*) and the OCI ingestor (`verify_kappa_axis`, `CC-10`) run where untrusted bytes enter; the threat modelled by a *forging store* that serves bytes which do not re-derive to the requested κ | the deployed Pages artifact is a *production target*: V&V proves Law L5 exhaustively in CI, so the deployed peer is not charged to re-prove it at runtime (ADR-019). A `ReadVerify` policy makes the placement explicit — `OnRead` re-derives κ (the default for a general/boundary caller, and the mode the conformance suite drives over *all* content) versus `Trusted`, which returns the canonical store's bytes as-is because the `KappaStore` *is* the memory and RAM is its cache (Law L3); the in-session browser peer reads locally with `Trusted` (`Console::resolve`). The receipt boundaries are unchanged and still verify every untrusted byte. Witnessed: the boundary read (`OnRead`) rejects a forging store's liar; the trusted read returns the store's bytes without re-deriving; honest content reads identically under both policies; `resolve_local` defaults to the verifying boundary policy | live (`vv/suites/cc29-read-verify-boundary`)
| CC-30 | A suspended machine resumes from its κ snapshot | the snapshot's own determinism (Law L1) and the reference RISC-V machine `qemu-system-riscv64` as the differential oracle | `Emulator::restore` is the exact inverse of `Emulator::snapshot` (ADR-009's *running state as a κ snapshot* property made whole), so suspend → resume is a round trip: `restore(snapshot(m))` re-snapshots to the *same* bytes and κ — including a machine with a virtio-blk disk — and the resumed machine continues execution *byte-identically* (same console, same final κ). Resume is κ-addressed content, so a suspended machine migrates by its κ alone (Law L1); a truncated snapshot is rejected, never half-restored. The strong witness: a real Linux machine *suspended mid-boot* and resumed from its κ reaches the byte-identical userspace the un-suspended machine did. This is the substrate primitive a second launch resumes from instead of cold-booting | live (`vv/suites/cc30-resume`)
| CC-31 | A devcontainer resumes from a persisted κ snapshot over OPFS, in the browser | the host `CC-30` differential-resume witness + the substrate's own `verify_kappa` re-derivation contract, in a real browser (Chromium via Playwright) — the same wasm GitHub Pages serves | the browser peer *suspends* the running devcontainer to a κ snapshot (`Workspace::suspend`), gzips it — most of guest RAM is zero, so a ~512 MiB machine compresses to a few MiB — and persists it (with its κ) to the Origin Private File System. Across a real page *reload* (a brand-new runtime; only OPFS survives), it reads the snapshot back, *verifies it by re-derivation* (Law L5 — OPFS is durable but untrusted storage, ADR-019; a tampered snapshot is refused and it cold-boots), and *resumes* the machine (`Workspace::resume_devcontainer`) — no kernel fetch, no rootfs assembly, no cold boot. Witnessed: the snapshot gzips to a small fraction of the machine; after reload it re-derives to its recorded κ; the *workspace files are intact*; the resumed machine is *live* and continues the very boot it was suspended partway through, to userspace; a tampered snapshot is rejected. The machine snapshot is *κ-pure*: it does not carry the console *output* buffer (a projection of the past, not future-affecting machine state), so a freshly restored machine's console is empty — the terminal layer, not the machine, restores the visible scrollback (persisted beside the snapshot and replayed on resume), so a resume at the *idle shell* (the steady state the periodic snapshot captures) comes back to the prompt you left, not a blank screen | live (`resume-test.mjs`; the host idle-shell resume witness `cc31_resume_terminal`)
| CC-32 | Distinct holospaces never bleed: each holospace's durable resume state is keyed by its identity (κ) | a deterministic keying witness that drives the real `build-workbench` config wiring and the real `holospace-fs` keying helpers (the deployed artifacts) | OPFS is per-origin, shared by every workbench tab, so a *fixed* snapshot filename would make every holospace read and write the *same* slot — launching one holospace would resume another's machine, and a removed/renamed holospace's remnants would bleed into the next launch. Identity is what-not-where (Law L1): the Platform Manager opens `workbench.html?id=<κ>`, `build-workbench` carries that κ into the workspace folder *authority* (`holospace://<κ>/workspace`), and `holospace-fs` keys every durable artifact (the resume snapshot, its κ, the scrollback) by it — so the *same* holospace resumes its own state across sessions while *distinct* holospaces map to *disjoint* OPFS slots and can never collide. Removing or terminating a holospace clears exactly its slot (the Manager's cleanup mapping is held identical to the workbench's, witnessed). Witnessed: two distinct κ launch URLs resolve, end-to-end, to disjoint snapshot files; an empty identity is the lone `default` slot; the Manager's delete-key matches the workbench's | live (`snapshot-keying-test.mjs`, the browser conformance job)
| CC-33 | A process inside the booted devcontainer is reachable from the workbench over an in-process substrate bridge | TCP and the guest's own listening socket, over the emulator's userspace NAT + ingress (`CC-16`/`CC-21`); the `qemu-system-riscv64` slirp network as the differential oracle for the NAT path | the browser workbench's extension host and the system emulator are one process (ADR-020), so reaching a server *inside* the guest is an in-process ingress connection into the emulator's own TCP stack, not a network round trip: the `Workspace` peer exposes `dial`/`send`/`recv`/`close` to a guest port, writes the client's bytes into the guest socket and reads the guest's bytes back, pumping the emulator between turns — the inward, in-process dual of the egress relay (ADR-014), with no relay, socket, or server (Law L4/ADR-001). The transport is peer-agnostic (Q6): the same devcontainer κ exposes the same listener through this bridge in the browser and through a real host socket on a native peer. Witnessed: a devcontainer boots a guest TCP listener (its `/init` starts a real server), the host bridge dials it and a byte stream round-trips faithfully both directions; a closed connection tears down; the bytes the guest received equal the bytes sent and vice-versa | live (`vv/suites/cc33-guest-bridge`)
| CC-34 | A remote extension host runs in the devcontainer OS and the workbench drives it over the bridge | the VS Code https://github.com/microsoft/vscode/blob/main/src/vs/platform/remote/common/remoteAgentConnection.ts[remote-server protocol] (the management + extension-host connection a Codespace uses) and the VS Code https://code.visualstudio.com/api[extension API]; a real *workspace/Node* extension as the activation authority, with a language server (`CC-18`) as the concrete client | the devcontainer OS runs the Node-based remote extension host (ADR-015) as a guest service on a port; the workbench's remote-authority resolver returns the `CC-33` in-process bridge as its connection (ADR-020), so the browser peer hosts the *remote* model — non-web extensions (rust-analyzer, the GitHub authentication provider, debuggers) activate in the OS where the holospace's files (`CC-15`), terminal (`CC-11`), and network (`CC-16`) are, exactly as in a Codespace. The same κ runs at native speed on a capable peer; in-browser it is correct but interpreter-bound (a documented performance difference, not a capability one — Q6/ADR-020). Witnessed (host-first, native speed, then in-browser): the guest remote-server comes up, the bridge speaks the remote-server handshake, and a workspace/Node extension *activates* in the guest's extension host (its contribution observable); `rust-lang.rust-analyzer`'s real extension activates and drives the rust-analyzer server as a guest process | staged (`vv/suites/cc34-remote-ext-host`)
|===

The capability set above is the **Codespaces/Gitpod parity** target (ADR-016): a repository becomes a ready, real development environment with the authentic VS Code workbench, extensions and their integrations, language intelligence and debugging, port-forwarded app previews, dev-container Features and lifecycle, and personalization — realized over the substrate with no server. ADR-016 maps each capability to its conformance row and substrate primitive so the model has no gaps against Codespaces/Gitpod.

The `CS` rows are witnessed green by every documentation build. The `CC` rows are each witnessed by a suite in `vv/suites/` that validates the component against its external authority — never against itself — by composing the https://github.com/Hologram-Technologies/hologram[hologram] substrate (consumed by reference, ADR-006), at the rev pinned in the workspace `Cargo.toml`. Adding a component without satisfying its `CC` row leaves holospaces incomplete by definition.

=== Quality gates and test tiers

The quality commitments above are enforced from the first commit by continuous integration (`.github/workflows/ci.yml`), which runs on every change:

* *V&V* — `vv/run.sh`: CS-1..CS-6 specification conformance (V1–V8) and the CC-1..CC-11 component suites (`vv/suites/`), each against its external authority.
* *Format* — `cargo fmt --check`.
* *Lints* — `cargo clippy` with warnings denied; the workspace forbids `unsafe_code` and warns on `missing_docs`.
* *Docs* — `cargo doc` with warnings denied (no broken links; every public item documented).
* *Unit* — `cargo test --lib`: a building block in isolation.
* *Integration* — `cargo test --test integration`: building blocks composed.
* *End-to-end* — `cargo test --test e2e`: whole operator flows (over the real substrate runtime and a real content-addressed gateway).
* *Portability* — the peer core builds for every supported environment: native, browser (`--target wasm32-unknown-unknown`), and bare-metal (`--no-default-features --target thumbv7em-none-eabi`, `no_std`). The same holospace κ boots on any peer (Chapter 7; quality goal Q6).
* *Browser* — `scripts/browser-manager-test.sh`: the Hologram Platform Manager (the browser peer) runs in headless Chromium (Playwright) — sign in, provision (both compute forms), view, resolve + verify by re-derivation (Law L5), a live `.holo`-engine differential (the executor compiled to wasm yields a κ identical to native — `CC-2`, RT2), and the execution surface — a κ-addressed Wasm code module is validated against the host-ABI surface, provisioned, **and booted as a container in-browser** through hologram's `wasmi` interpreter engine (suspended to a κ snapshot, resumed, terminated — `CC-6`, ADR-008's contract generalized by ADR-009; an ambient import is refused). It is deployed to GitHub Pages by `.github/workflows/pages.yml` (the cold-start, untrusted gateway of Chapter 6).
* *Execution surface* — `cargo test --test cc6_execution_surface`: a κ-addressed Wasm code module (the surface the system emulator binds; ADR-008's contract, generalized by ADR-009) is validated against the WebAssembly specification and the substrate host-ABI surface, then booted, suspended, resumed, and migrated on the real `ContainerRuntime` over **both** engines — the native Wasmtime JIT and hologram's `wasmi` interpreter (the browser + bare-metal engine) — yielding the same holospace κ on each (Q6). A devcontainer holospace boots the code its config selects (`cargo test --test e2e`). The RT1 surface (`CC-6`).

Every `CC` suite and test tier witnesses its component against its external authority on every change; each runs from the first commit.
