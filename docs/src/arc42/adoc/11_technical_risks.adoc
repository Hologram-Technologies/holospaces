[[section-technical-risks]]
== Risks and Technical Debts

[options="header"]
|===
| # | Risk / debt | Mitigation
| RT1 | *Arbitrary-OS execution surface (resolved).* A devcontainer holospace runs an *arbitrary* operating system (a real Linux dev environment, real binaries) on a Wasm-only, no-WASI substrate — full Codespaces/Gitpod parity. | *Resolved as ADR-009* (superseding ADR-008): a κ-addressed *system-emulator codemodule* computes an arbitrary OS image — the image and repository as κ-addressed content, console/input/network as hologram channels, running state as a κ snapshot, imported and verified trustlessly (`get_with_fetch`), on the κ-addressed Wasm code-module surface (the Execution Surface building block) that runs on the per-environment `ContainerEngine` (Wasmtime native; `wasmi` interpreter in browser/bare-metal). Witnessed against the https://www.kernel.org[Linux] and https://containers.dev[Dev Container] specifications by the execution conformance rows of Chapter 10.
| RT2 | *Browser execution (realized).* Running the https://github.com/Hologram-Technologies/hologram[hologram] executor and a Wasm `ContainerEngine` in the browser (compiled to Wasm via `wasm-pack`) was the keystone build. | *Realized:* the `.holo` executor runs in headless Chromium and produces a κ identical to the native executor for the same `.holo` (`CC-2`), **and** the browser peer boots Wasm code modules in-browser — including the system-emulator codemodule — via hologram's `wasmi` interpreter `ContainerEngine` (`CC-6`), witnessed live by `scripts/browser-manager-test.sh` (the CI `browser` job).
| RT3 | *Documentation toolchain weight.* The doc pipeline requires JDK 21, Docker, Ruby, a large Structurizr download, and a built cmark-gfm; it cannot run in a base container without provisioning. | Pinned in the devcontainer and `docs/scripts/install-tools.sh`; exercised in CI (`.github/workflows/ci.yml`). The JDK is pinned to 21 to match the pipeline.
| RT4 | *Cross-peer rendezvous.* Discovering an operator's other instances without a server depends on hologram's content-addressed networking. | Use hologram's mechanism (KappaSync) as-is; do not introduce a signaling server or any host/peer-id naming (Law L1).
|===

This chapter is maintained as risks are resolved (becoming decisions in Chapter 9) or retired.
