[[section-glossary]]
== Glossary

[options="header"]
|===
| Term | Definition
| *κ-label (kappa-label)* | A content address, `<axis>:<hex>` = `H(canonical_form)`, supplied by https://github.com/UOR-Foundation/uor-addr[UOR-ADDR]. The identity of every thing in holospaces (_what_, not _where_).
| *Canonical form* | Deterministic bytes addressed by a κ-label; the only form holospaces operates on.
| *holospace* | A bootable, κ-addressed environment — the unit holospaces provisions, runs, and manages.
| *holo-file* | A `.holo` compute artifact; one way to provision a holospace (run via the hologram executor).
| *devcontainer* | A repository + `devcontainer.json` per the https://containers.dev[Dev Container] specification — the definition of a development environment (base OS image, tools, config).
| *devcontainer holospace* | A holospace provisioned from a devcontainer; its operating system is computed by the System Emulator from κ-addressed content (a Codespaces/Gitpod-like environment).
| *peer* | An environment (browser / native / bare-metal) that becomes the substrate by realizing it locally.
| *Operator* | The human who signs in and provisions/manages holospaces through the Manager.
| *Hologram (the platform)* | holospaces' first-party holospace; its code is the Platform Manager.
| *Platform Manager* | The operator console that manages holospaces; served from GitHub Pages.
| *Sign-in / identity* | Unlocking a self-sovereign key (not a server account) that links an operator's instances for sync.
| *The substrate* | The https://github.com/Hologram-Technologies/hologram[hologram] storage (KappaStore) · network (KappaSync) · runtime (ContainerRuntime), plus the `.holo` executor. Authoritative in hologram.
| *Two compute forms* | A Wasm code module (general/system code) or a tensor `.holo` (compute); both κ-addressed.
| *Verify by re-derivation* | Accepting a received byte only after re-computing its κ.
| *The store is the memory* | The content-addressed store is the address space; RAM is a cache.
| *κ snapshot* | A holospace's running state captured as a κ-addressed canonical form; the unit of suspend / resume / migrate.
| *Capabilities* | The κ-addressed authority a holospace is spawned with (storage roots, channels, budgets); part of its reproducible definition. The capability model is hologram's.
| *Boot Layer* | holospaces' environment-agnostic core that resolves a holospace κ and spawns it through the substrate runtime (Chapter 5).
| *Realizations* | holospaces' canonical-form types — chiefly the holospace — κ-addressed and verified by re-derivation.
| *ContainerEngine* | The backend seam hologram's `ContainerRuntime` drives to run a Wasm code module — Wasmtime natively, the `wasmi` interpreter in the browser and on bare-metal. Defined by hologram. (The `.holo` engine is a *distinct compute path*, not a ContainerEngine.)
| *Execution Surface* | The κ-addressed Wasm code-module contract a holospace's code binds: it imports only the `hologram` host ABI and presents the container ABI. Defined and enforced by holospaces (Chapter 5); the substrate's `ContainerRuntime` boots it.
| *codemodule* | Code addressed by a κ-label (content, not a location). Drivers, container bodies, and the system emulator are codemodules — imported by κ, verified by re-derivation, and instantiated.
| *System Emulator* | The execution codemodule for a general operating system (ADR-009): a system emulator compiled to Wasm over the host ABI that computes an *arbitrary* OS image from κ-addressed content. Starts with Linux. Two ISA targets (ADR-021): RV64GC and AArch64.
| *Guest architecture (ISA)* | The instruction set a holospace's guest runs — `riscv64` (RV64GC) or `aarch64` (ARMv8-A). The operator selects it in the Manager before provisioning; it is part of the holospace's content-addressed identity, so it is *fixed for the holospace's lifetime* (changing it mints a different holospace). Other guest parameters stay reconfigurable at runtime (`CC-28`).
| *AArch64 (ARMv8-A)* | ARM's 64-bit architecture (the A64 instruction set). holospaces' second emulator ISA target: the A64 core + the EL0/EL1 exception model, VMSAv8-64 paging, and the ARM `virt` platform (GICv2, the generic timer, PSCI, a PL011 console), booting a real `arm64` Linux verified against `qemu-system-aarch64` (`CC-35`/`CC-36`/`CC-37`).
| *κ-disk* | A `KappaStore`-backed block device: the operating-system image and repository held as κ-addressed blocks; the System Emulator's storage.
| *Projection* | A surface that renders and drives a holospace's canonical content — a view + intent surface, never a server or a second source of truth (Law L3). The Platform Manager is the management projection.
| *Workspace Projection* | A projection that gives the Codespaces/Gitpod experience over a *running* holospace: a browser editor, file tree, and terminal that read the environment's content by κ and publish operator input as canonical events on its channels.
| *Channel* | A κ-addressed `publish` / `subscribe` route on the substrate; a holospace's console, input, and network are canonical events on channels.
| *Canonical event* | A κ-addressed message published on a channel (e.g. an operator keystroke or a console byte) — the uor-native form of I/O.
| *Host ABI* | The closed `hologram` import surface a Wasm code module may call (`storage_put`/`get`, `publish`/`subscribe`, …); no WASI or ambient host (Law L1/L4).
| *The laws* | The five invariants of Chapter 2: L1 identity is content not location; L2 canonical forms only; L3 the store is the memory; L4 everything through the substrate; L5 verify by re-derivation.
|===
