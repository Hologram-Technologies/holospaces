# Imported validation artifacts — provenance

Every artifact holospaces is validated against is external and authoritative, imported and pinned. This file is the authoritative record of *what* is imported, *from where*, at *which pin*, and *how it is verified*. See the Conformance catalog in [arc42 chapter 10](../docs/src/arc42/adoc/10_quality_requirements.adoc) for which invariant each artifact validates.

## Live (component conformance, `CC-*`)

Every component is implemented as a thin composition of the [hologram](https://github.com/Hologram-Technologies/hologram) substrate, consumed by reference (ADR-006) at the pinned rev `18f553d8578997ce32e7b653786a0bcf9b09a2c0` (git dependency, see the root `Cargo.toml`). Each `CC-*` row is witnessed against an external authority by its suite in `vv/suites/`, run by `vv/run.sh`.

Every artifact is external/authoritative — imported verbatim and pinned, never hand-authored. (This mirrors hologram's own conformance discipline, e.g. its `AS` class validates the σ-axis against the reference `blake3` crate, not against a vendored vector file.)

### `CC-9` (in progress) — the system emulator, grown conformance-first

CC-9's end state — *a real operating system boots and runs on the emulator* — is reached against the **RISC-V ISA** as the external authority, exactly as `CC-5` is grown against the WebAssembly spec suite. The first step is **witnessed now in the cargo test tier** (`tests/cc9_emulator.rs`, run by CI), and is *not yet* wired as a `vv/suites/cc9-*` suite — so `vv/run.sh` honestly reports **CC-9 pending** until a real OS boots.

| Step | Authority | Source / pin | Verified by |
|---|---|---|---|
| `CC-9.a` emulator ISA conformance | the **RISC-V Unprivileged ISA** (RV64IMA + Zicsr), per <https://riscv.org/technical/specifications/>; real machine code assembled by **LLVM's RISC-V backend** (the external assembler), in the self-checking style of <https://github.com/riscv-software-src/riscv-tests> | `vv/artifacts/cc9/*.bin` — flat programs assembled by `llvm-mc --triple=riscv64 --mattr=+m,+a,+zicsr`/`llvm-objcopy` from the committed `*.s` (base integer, M, A atomics, Zicsr CSRs); expected ISA results in `expected.txt`; `sha256` in `bins.sha256`; command + LLVM version in `vv/artifacts/cc9/SOURCE.txt` | `tests/cc9_emulator.rs`: the `emulator::Emulator` (RV64IMA + Zicsr) runs each real program and yields the ISA-defined result exactly; the `write` syscall produces real console output; the machine's κ snapshot verifies by re-derivation (L5) and is reproducible across runs and peers (L1), read off a `CC-7` κ-disk |
| `CC-9.b` emulator on the substrate | the **hologram `ContainerRuntime` contract** + the **WebAssembly spec / host-ABI surface** (`CC-6`) — the real Wasmtime engine as the execution oracle | `vv/artifacts/cc9/emulator.wasm` — the emulator core compiled to a hologram container codemodule (crate `holospaces-emulator`; imports only `hologram.storage_put`, exports the container ABI); built by `scripts/build-emulator.sh`, `sha256` in `bins.sha256`, command in `SOURCE.txt` | `tests/cc9_emulator.rs`: the codemodule validates against the execution-surface contract (`surface::validate_userland`), then is **spawned on the real Wasmtime `Runtime`** with a guest image as its initial state; it runs the real RISC-V program and emits the ISA-defined result into the substrate via the host ABI (content-addressed — the result κ is the guest's deterministic output); the container κ snapshot is reproducible. The emulator is κ-addressed Wasm over the host ABI, not a parallel medium (Law L4) |

Remaining CC-9 steps (subsequent sessions): the privileged architecture (CSRs, traps, Sv39 paging, CLINT/PLIC, SBI), the console over `publish`/`subscribe` channels, booting a real Linux image to userspace, and the QEMU differential oracle — at which point the `vv/suites/cc9-emulator-os` suite goes live and CC-9 flips to witnessed.
