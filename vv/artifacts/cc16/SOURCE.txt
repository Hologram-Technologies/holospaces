CC-16 — VirtIO networking authority artifacts.

kernel/Image.gz — a real Linux 6.6.0 RV64GC kernel built from defconfig with
  VIRTIO_NET + VIRTIO_BLK + VIRTIO_MMIO + EXT4 + NET_9P_VIRTIO/9P_FS, the INET
  IPv4 stack, the kernel's built-in DHCP autoconfiguration (IP_PNP + IP_PNP_DHCP,
  used via the `ip=dhcp` kernel cmdline), and an SBI console. Reproducible build
  (KBUILD_BUILD_TIMESTAMP/USER/HOST fixed); kernel.config committed. Toolchain:
  riscv64-linux-gnu-gcc.

initnet.c — a freestanding RV64 program (no libc, raw syscalls): after the
  kernel does DHCP, it socket()s, connect()s a TCP stream (to 10.0.2.9:8080),
  sends an HTTP request, reads the response (NET-CONNECTED / NET-RECV / NET-DONE),
  and powers off. The guest userspace that exercises the network path.

image/ — a real OCI image-layout whose tar+gzip layer is /init = initnet
  (compiled from initnet.c with riscv64-linux-gnu-gcc -nostdlib -static -Os
  -ffreestanding -fno-builtin, stripped; reproducible tar: numeric owner, fixed
  mtime; gzip -9 -n). The witness (crates/holospaces/tests/cc16_network.rs)
  ingests this image (CC-10), assembles it into an ext4 rootfs (CC-7), and boots
  it on the emulator with networking attached (boot_net + a native StdEgress
  port-forwarding 10.0.2.9:8080 → 127.0.0.1:PORT, the analogue of QEMU guestfwd).

Differential oracle — qemu-system-riscv64's user-mode (slirp) network is the
  reference holospaces' userspace NAT must match. Confirmed: the guest does DHCP
  ("IP-Config: Complete"), opens a TCP connection through the NAT, and completes
  an HTTP exchange to a host server:
    python3 -m http.server (a host listener on 127.0.0.1:PORT), then
    qemu-system-riscv64 -M virt -m 512M -nographic -bios default \
      -global virtio-mmio.force-legacy=false -kernel <Image> \
      -drive file=<rootfs.ext4>,format=raw,if=none,id=hd0 -device virtio-blk-device,drive=hd0 \
      -netdev user,id=n0,guestfwd=tcp:10.0.2.9:8080-tcp:127.0.0.1:PORT \
      -device virtio-net-device,netdev=n0 \
      -append "root=/dev/vda rw console=hvc0 earlycon=sbi ip=dhcp init=/init"
  → IP-Config: Complete · NET-CONNECTED · NET-RECV: HTTP/1.0 200 OK … · NET-DONE.

The emulator-side virtio-net device + the userspace TCP/IP NAT (ARP + DHCP + the
guest-facing TCP state machine) and the native egress transport (a direct host
socket) are IMPLEMENTED and witnessed against this oracle
(crates/holospaces/src/emulator/net.rs; crates/holospaces/tests/cc16_network.rs;
vv/suites/cc16-network.sh). The NAT is no_std and compiles to wasm32 with the
peer core; the browser peer substitutes a WebSocket tunnel to a relay for the
egress seam (ADR-014). That browser slice is ALSO IMPLEMENTED and witnessed:
crates/holospaces-web/src/wsnet.rs (the WsEgress over a WebSocket),
crates/holospaces-web/web/relay.mjs (a pure-Node RFC 6455 TCP-over-WebSocket
relay — the egress gateway), and crates/holospaces-web/web/devcontainer-net-test.mjs
(Chromium/Playwright: the same wasm peer boots the net devcontainer in a tab,
does DHCP over virtio-net, and completes the HTTP exchange tunnelled over the
relay) — wired into scripts/browser-manager-test.sh.
