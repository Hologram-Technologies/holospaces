CC-17 — the real VS Code web workbench (authority artifacts).

The workspace is the REAL VS Code web workbench — the same compilation that runs
vscode.dev / github.dev (ADR-012), not a reconstruction from components (that is
CC-13's Monaco + xterm). holospaces embeds it as the Workspace Projection and,
per ADR-015, drives a remote extension host inside the devcontainer OS so
extensions and integrations work as in a Codespace.

Pin (the authority):
  npm package  vscode-web@1.91.1  (a packaged build of the official Microsoft
    VS Code web compilation; https://www.npmjs.com/package/vscode-web)
  npm integrity (sha512 of the tarball):
    sha512-kzrbnxxmAmBDBLcTXQBM5Ehucdf35MI4pX1DHMPa6SFFg4IVKd18Qn6TrdG+VT4O4fk6nySovNP22GHky3nD8Q==
  tarball: https://registry.npmjs.org/vscode-web/-/vscode-web-1.91.1.tgz

vendor.sha256 — the sha256 of the workbench's EXECUTABLE CORE (the AMD loader,
  the web package paths, the workbench.web.main bundle + its nls, and the
  workbench.js bootstrap). The browser peer re-derives each file's sha256 over
  the served bytes and REFUSES to load on any mismatch (Law L5) — the workbench
  is verified content, exactly as CC-13 verifies Monaco + xterm by re-derivation.
  The 60 MB build itself is not committed (it is fetched by the pin above, as the
  browser tests fetch Playwright by pin); the manifest is its κ-identity.

Witness — crates/holospaces-web/web/vscode-workbench-test.mjs (Chromium via
  Playwright): assembles the pinned vscode-web dist, VERIFIES the executable core
  against vendor.sha256 (a forged byte is refused), serves it as the holospaces
  workspace tab, and asserts the REAL workbench boots to its authentic UI (the
  "holospaces VS Code" title, the .monaco-workbench, the activity bar).

Phase status (this row is staged; ADR-012/015):
  Phase 1 (DONE/witnessed) — the real workbench loads κ-verified in the tab.
  Phase 2 FOUNDATION (DONE/witnessed) — the editor-side filesystem over the
    shared workspace: holospaces' list/write/read API over the virtio-9p
    workspace (CC-15), content-addressed, with a file the editor writes read by
    the real OS over 9p (one content, Law L1). Witness
    crates/holospaces/tests/cc17_workspace_fs.rs; exposed on the wasm peer as
    Workspace.ws_list/ws_read/ws_write for the workbench's FileSystemProvider.
  Phase 2 SUPPORTED-MECHANISM WITNESS (DONE) — the real VS Code web workbench is
    served the holospace's workspace the way github.dev / vscode.dev / Codespaces
    do: Microsoft's own @vscode/test-web built-in FileSystemProvider (--browser
    none serves; our own Chromium drives, as the harness's bundled launcher is
    unavailable here). The genuine workbench loads and its readDirectory reaches
    the real editor. Witness crates/holospaces-web/web/vscode-workbench-fs-test.mjs;
    suite vv/suites/cc17-workbench-fs.sh. NO hand-rolled embedder / service worker
    / custom extension — the earlier attempts at those were the legacy detour.
  Phase 2 (remaining) — wire the same supported FileSystemProvider serving to the
    LIVE wasm peer's ws_* (the running 9p workspace) in the holospaces page, and
    the terminal over the console (CC-11). The extension host is per-peer
    (ADR-015 feasibility refinement): the BROWSER peer uses the vscode.dev "web"
    model (a FileSystemProvider web extension whose fetches a service worker
    serves from the wasm peer's 9p workspace), because its emulated OS does not
    host the Node-based vscode-server; a server-capable (native/remote) peer uses
    the "remote" model (the extension host in the OS over the remote protocol).
  Phase 3 — extension integrations + LSP/DAP (CC-18/CC-19) on whichever host.
  NB (a witnessed finding): a custom builtin FileSystemProvider web extension
    served over HTTP via additionalBuiltinExtensions did NOT activate under
    headless Chromium in vscode-web@1.91.1 (the web-worker extension host starts
    but the extension's code is never loaded; webEndpointUrlTemplate same-origin
    + package.nls.json + "*" activation did not fix it). The Phase-2 web model
    needs that activation path resolved (likely the service-worker /
    extension-resource loading), so it is not yet witnessed.
